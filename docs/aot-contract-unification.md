# Unifying the AOT instrumentation contract on TypeSpec

Goal: one source of truth (TypeSpec) for the .NET AOT instrumentation contract, emitting both
the C# the source generator consumes and the JSON the verifier consumes — so the Python
generator and the hand-owned C# both disappear. This is the "kill the Python in a C# project"
end state, done correctly rather than by freezing a copy.

This is an **attended** migration (it changes the autoinstrumentation build). Do it on a branch,
keep `bash tools/smoketest.sh` green at every step, never on main unverified.

## Why this is now the right next step

`@ancplua/typespec-emit-csharp` already generates C# from TypeSpec today
(`tsp compile` → `generated/contracts/Qyl/Api/Contracts.cs`). TypeSpec → C# is a first-class
emitter path; no TypeScript detour is needed.

Honest current state (after commit `a359782`, which removed Python's C# generation):

- `qyl-dotnet-autoinstrumentation/src/.../InstrumentationContract.cs` is now **owned C#**
  (internal enums `InstrumentationContractKind | Signal | Lane | QylContractStatus |
  SourceVisibility | PayloadAccess`, plus the 60-item contract data), consumed at build time by
  `QylAutoInstrumentationGenerator.cs`.
- The YAML SSOT (`docs/contracts/otel-dotnet-auto-60.upstream.yaml` + `qyl-aot-ownership.yaml`
  → `qyl-aot-contract.resolved.yaml`) still feeds the Python generator for the JSON/YAML/schema
  data artifacts (`conformance-plan.json`, coverage matrix, contract schema, README table).

→ The cut traded "Python renders C#" for "C# is owned but decoupled from the YAML." Both are in
sync now (frozen correctly), but that is a drift seam: edit the YAML and the C# no longer
follows. The unification closes it by making **one** TypeSpec model the source of both.

## Target shape

```text
api-schema:  models/aot-contract.tsp   (60 items + enums, the SSOT)
                      │
        ┌─────────────┴─────────────┐
   csharp emitter               json/control-graph emitter
        │                             │
  InstrumentationContract.cs    conformance-plan.json + matrix + schema
  (into autoinstrumentation)    (verifier inputs)
```

Python `generate-contract-artifacts.py` is deleted. The hand-owned `InstrumentationContract.cs`
becomes generated again — but from TypeSpec, not Python. One SSOT, one toolchain (Node).

## Steps (each ends green)

1. **Model the contract in TypeSpec.** In `qyl-api-schema`, add `models/aot-contract.tsp`: the six
   enums above and a `ContractItem` model with the fields the resolved YAML carries (`key`,
   `upstream_source`, `lane`, `qyl_status`, `call_site_visibility`, `payload_access`, `evidence`,
   `conformance_signals`). Port the 60 items as instances. Validate: `npx tsp compile main.tsp` green.
2. **Teach the C# emitter the runtime shape.** `@ancplua/typespec-emit-csharp` must emit
   `InstrumentationContract.cs` byte-equivalent to today's owned file (same `internal` enums + the
   `ImmutableArray<ContractItem>` data + `TotalCount`). Diff the emitted output against the current
   owned file until identical. This is the load-bearing step — the source generator depends on the
   exact shape.
3. **Wire generation into autoinstrumentation.** Point the build at the emitted file (output path
   `src/Qyl.AutoInstrumentation.SourceGenerators/InstrumentationContract.cs`). Re-add an
   `// <auto-generated/>` header now that it is generated again. `dotnet build` the SourceGenerators
   project green.
4. **Move the JSON artifacts.** Have the control-graph (or a sibling) emitter produce
   `conformance-plan.json` in the pinned wire shape `{schema_version, graph_schema_version,
   services:[{service_name, profile_id, expected_signals:[{kind, name, required_attributes,
   recommended_attributes, opt_in_attributes}]}]}`. Diff against the current Python output.
5. **Delete the Python + the YAML duplication.** Once C# and JSON both come from TypeSpec and diff
   clean, delete `tools/generate-contract-artifacts.py` and the `docs/contracts/*.yaml` +
   `docs/generated/*` it owned, and the verify scripts that gated regeneration. Keep the YAML only
   if a non-TypeSpec consumer still needs it (none known).
6. **Gate.** `bash tools/smoketest.sh` (pack + consumer build) and the snapshot/verifier suite must
   stay green. Only then merge.

## The one hard risk

Step 2. The source generator consumes `InstrumentationContract` by exact member shape. If the
emitted C# diverges (a renamed field, a different collection type, a missing `TotalCount`), the
autoinstrumentation build breaks — and two hackathon projects depend on that build. Drive step 2
by **byte-diff against the current owned file**, not by eyeballing. Until the diff is clean, the
owned file stays authoritative.

## Smaller safe first slice (if you want proof before committing)

Model the contract in TypeSpec (step 1) and emit it to a **scratch path**, then byte-diff against
the current `InstrumentationContract.cs`. Zero diff = the migration is proven and step 3 is a
one-line path change. Non-zero diff = you see exactly what the emitter must fix, with no risk to
the live build.
